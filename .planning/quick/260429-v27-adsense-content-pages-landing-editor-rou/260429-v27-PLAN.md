3. **섹션 1: 이메일 HTML의 기본 원칙** (h2)
   - 이메일 HTML은 웹 HTML과 다릅니다
   - 테이블 기반 레이아웃을 사용하는 이유
   - DOCTYPE과 기본 구조
   - 600px 최대 너비의 중요성
   - 콘텐츠: "이메일 HTML은 웹 HTML과 근본적으로 다릅니다. 최신 웹 브라우저는 CSS Grid, Flexbox, 시맨틱 태그를 지원하지만, 이메일 클라이언트는 각각 다른 렌더링 엔진을 사용합니다. Outlook Classic은 Microsoft Word의 렌더링 엔진을 사용하고, 네이버 메일은 `<style>` 블록을 완전히 제거합니다. 따라서 이메일 HTML은 가장 낮은 공통분모에 맞춰 작성해야 합니다. `<table>` 기반 레이아웃을 사용하는 것이 여전히 가장 안전한 방법입니다. `<div>` 레이아웃은 일부 클라이언트에서 깨질 수 있습니다. 이메일의 최대 너비는 600px로 제한하세요. 대부분의 이메일 클라이언트의 읽기 영역이 이 너비에 최적화되어 있습니다."

4. **섹션 2: 클라이언트별 CSS 호환성 팁** (h2)
   - 네이버 메일: `<style>` 블록 완전 제거, margin/padding/font-family 인라인도 차단. 해결법: 셀 패딩으로 간격 조절, 웹폰트 대신 시스템 폰트 사용.
   - Gmail: 조건부 `<style>` 제거 (background-image url(), @import, @charset 감지 시 전체 블록 제거). 해결법: background-image는 인라인으로, @import 사용 금지.
   - 다음/카카오: script, iframe, object, embed 요소 제거. form 요소 주의. 해결법: 인터랙티브 요소 대신 이미지+링크 조합 사용.
   - Outlook Classic: Word 엔진 사용. background-image, border-radius, CSS animation 미지원. 해결법: VML을 사용한 배경 이미지, 이미지로 둥근 버튼 대체.
   - Outlook New: 웹 기반으로 Gmail과 유사한 제한. 단, max-width 지원.

5. **섹션 3: 인라인 CSS를 사용하는 이유** (h2)
   - `<style>` 블록이 제거되는 클라이언트 목록
   - 인라인 CSS의 장점과 단점
   - 자동 인라이닝 도구 (Sticky의 "Inline CSS" 기능 소개)
   - 콘텐츠: "네이버 메일은 `<style>` 블록을 무조건 제거합니다. Gmail도 특정 조건에서 `<style>` 블록을 제거합니다. 따라서 중요한 스타일은 반드시 각 HTML 요소의 style 속성에 직접 작성해야 합니다. 이것이 '인라인 CSS'입니다. 수동으로 인라인하면 유지보수가 어렵기 때문에 Sticky의 'Inline CSS' 버튼을 사용하면 `<style>` 블록을 자동으로 인라인 스타일로 변환할 수 있습니다."

6. **섹션 4: 반응형 이메일 작성 팁** (h2)
   - 미디어 쿼리의 한계 (Gmail은 미지원)
   - 유동적 레이아웃 (percent 기반 너비)
   - 하이브리드 코딩 기법
   - 모바일 우선 vs 데스크톱 우선
   - 콘텐츠: "반응형 이메일은 @media 쿼리에 의존하지만, Gmail 앱은 미디어 쿼리를 지원하지 않습니다. 따라서 '하이브리드 코딩' 기법을 사용합니다: 기본 레이아웃을 모바일 크기(100%)로 설정하고, 미디어 쿼리로 데스크톱 레이아웃을 추가합니다. max-width와 width를 함께 사용하면 미디어 쿼리 없이도 유연한 레이아웃을 만들 수 있습니다."

7. **섹션 5: 자주 하는 실수와 해결법** (h2)
   - 5-6개 항목, 각각 "실수"와 "해결법" 쌍으로 구성
   - 실수 1: div 기반 레이아웃 사용 → 해결: table 레이아웃
