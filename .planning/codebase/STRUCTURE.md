## Directory Purposes

**app/:**
- Purpose: Contains all application pages, layouts, components, and styling
- Contains: Route definitions via file structure, React components, global CSS
- Key files: `layout.tsx` (root layout), `page.tsx` (homepage), `globals.css` (styling)
- Pattern: Next.js App Router - file paths map directly to URL routes

**public/:**
- Purpose: Static assets served directly at root URL without processing
- Contains: SVG icons, images, fonts, static files
- Key files: `next.svg`, `vercel.svg`, favicon.ico
- Pattern: Files in public/ accessible as `/filename` in HTML/CSS

**.next/:**
- Purpose: Build output directory (generated by `next build`)
- Contains: Compiled code, optimized bundles, type definitions
- Generated: Yes (created during build, not in source control)
- Committed: No (included in .gitignore)

**.planning/codebase/:**
- Purpose: GSD codebase analysis and mapping documents
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, etc.
- Generated: No (manually created by codebase mapper)
- Committed: Yes (part of project documentation)

## Key File Locations

**Entry Points:**
- `app/page.tsx`: Homepage route component - renders `/` path
- `app/layout.tsx`: Root layout - wraps all pages with shared structure
- `next.config.ts`: Next.js build and runtime configuration

**Configuration:**
- `tsconfig.json`: TypeScript compiler options (strict mode, path aliases)
- `eslint.config.mjs`: Code quality linting rules (Web Vitals, TypeScript)
- `postcss.config.mjs`: PostCSS plugin configuration (Tailwind CSS v4)
- `package.json`: Dependencies and npm scripts (dev, build, start, lint)

**Core Logic:**
- `app/page.tsx`: Homepage component logic and JSX rendering
- `app/layout.tsx`: Root layout logic, font loading, metadata setup
- `app/globals.css`: Global styles, CSS variables, Tailwind imports

**Testing:**
- Not configured - no test files present in codebase
- ESLint provides static code quality checks

**Styling:**
- `app/globals.css`: All global CSS, Tailwind imports, CSS custom properties
- Inline Tailwind classes in component files via className attributes

## Naming Conventions

**Files:**
- `layout.tsx`: Layout components using Next.js App Router convention
- `page.tsx`: Route pages using Next.js App Router convention
- `*.tsx`: React components written in TypeScript
- `*.css`: CSS stylesheets (globals.css for global scope)
- `*.config.*`: Configuration files (next.config.ts, postcss.config.mjs, etc.)
- `.mjs`: ES Modules configuration files (postcss, eslint)

**Directories:**
- `app/`: Lowercase, matches Next.js convention
- `public/`: Lowercase static assets directory
- Directories in `app/` that become routes (not yet used - would be lowercase, kebab-case for multi-word routes)

**Components:**
- PascalCase for React components (RootLayout, Home)
- Server Components by default unless "use client" directive added
- No suffix convention currently (would follow -Server, -Client patterns if needed)

## Where to Add New Code

**New Feature (Full-Stack Page):**
- Primary code: Create new `app/[route]/page.tsx` for route
- Styles: Add global styles to `app/globals.css` OR create `app/[route]/page.css`
- Tests: Not yet configured - would go in `tests/` directory when testing is added

**New Component (Reusable):**
- Implementation: Create in new `app/components/` directory (does not become a route)
- File: `app/components/ComponentName.tsx` (PascalCase)
- Styles: Inline Tailwind classes or create `app/components/ComponentName.css`
- Pattern: Export as named export or default export

**New Route/Page:**
- Implementation: Create `app/routename/page.tsx`
- Layout: If route needs unique layout, create `app/routename/layout.tsx`
- Example: Creating `/about` route → create `app/about/page.tsx`

**Utilities/Helpers:**
- Shared helpers: Create `app/lib/` directory for utility functions
- Export: Use default or named exports
- Pattern: `app/lib/helpers.ts` → `import { helperFunction } from "@/lib/helpers"`

**Styling:**
- Global: Modify `app/globals.css`
- Component-scoped: Use inline Tailwind classes via className prop
- CSS Modules: Not yet used - would go in `app/` alongside components as `*.module.css`

## Special Directories

**build (/):**
- Purpose: Production-ready compiled output
- Generated: Yes (created by `next build`)
- Committed: No (in .gitignore)

**out (/):**
- Purpose: Static export output (if static export enabled in next.config.ts)
- Generated: Yes (if using `next export`)
- Committed: No (in .gitignore)

**coverage (/):**
- Purpose: Test coverage reports
- Generated: Yes (by test runner)
- Committed: No (in .gitignore)

**node_modules (/):**
- Purpose: Installed npm/pnpm dependencies
- Generated: Yes (by package manager)
- Committed: No (in .gitignore)

**pnpm-lock.yaml:**
- Purpose: Lock file for reproducible pnpm installs
- Generated: Yes (by pnpm)
- Committed: Yes (enables exact reproduction of dependency versions)

## Path Aliases

**Configuration:**
- Defined in `tsconfig.json`: `"@/*": ["./*"]`
- Usage: `import { Component } from "@/app/components/Component"`
- Benefit: Absolute imports work anywhere in project without relative paths
